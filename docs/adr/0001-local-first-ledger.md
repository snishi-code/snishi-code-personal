# ADR 0001: ローカルファースト家計簿（IndexedDB 主体 + JSON 交換）

- ステータス: 採用
- 日付: 2026-06-06
- 対象: `simple-ledger`（個人カテゴリ）

## 背景

旧版は GAS + Google Spreadsheet で、入力のたびに Google サーバーへ通信していた。
サイト憲法は **ユーザーデータの外部送信を絶対禁止（全カテゴリ例外なし）** と定める。
家計簿アプリを配信するには、この前提でゼロから設計し直す必要があった。

## 決定

1. **実行時の正本は端末内 IndexedDB**。アプリは外部送信系（`fetch` 等）を一切持たない。
2. **端末間共有・バックアップは JSON export/import** を公式交換形式とする。
3. **アプリ内に同期・送信機能を持たない**。同期はアプリ外手段（Obsidian Sync 等）に委ねる。
4. 会計モデルは複式簿記（借方/貸方）。PL/BS は保存せず仕訳から毎回導出する。

## 根拠

### なぜ IndexedDB 主体か（JSON ファイルを常時 DB 代わりにしない）

- 構造化クエリ・インデックス・トランザクション（原子的な置換）が必要。JSON ファイルを
  直接読み書きし続ける設計は、原子性が弱く、肥大化・破損・競合に弱い。
- IndexedDB は容量・耐久性・同一オリジン隔離の面でブラウザの正規ストレージ。
- JSON は「スナップショット可能な交換形式」として最適だが、**実行時ストアには向かない**。
  そこで「DB = 正本 / JSON = 交換形式」と役割を分離した。

### なぜアプリ内同期を持たないか

- どんな同期もサーバー往復＝外部送信を伴い、憲法（外部送信ゼロ・例外文を作らない）に反する。
- 「送信可」の例外を 1 つでも作ると、その例外文自体が CLAUDE.md / メモリ経由の漏洩源になる。
- ユーザーが本当に共有したい場合は、export した JSON をユーザー自身が選んだ手段
  （Obsidian Sync、AirDrop、クラウドドライブ等）で運ぶ。アプリはその手段を操作しない。

### import の安全性

自動マージは行わず、**fail-closed** とする（[ledger-protocol.md](../dev/ledger-protocol.md)）。
Zod 検証 → スキーマ版確認 → import 前スナップショット → 原子的置換。revision 不一致は
自動上書きせずユーザー確認を求める。

## 結果（トレードオフ）

- 端末間のリアルタイム同期は無い（手動の export/import）。これは憲法と引き換えに受け入れる。
- オフラインで完結し、プライバシーが最大化される。
- 将来「送信ありの機能」が本当に必要になった場合は、サイトの約束を濁さないよう **別途設計を相談**する
  （安易に例外文を作らない）。

## 将来ネイティブアプリへ移植する場合の考え方

- ドメイン層（`src/domain/`：型・複式ロジック・PL/BS 導出・スキーマ・migration）は UI/ストア非依存
  なので、ネイティブ（例: SQLite + 同一の JSON 交換形式）へそのまま移植できる。
- `LedgerExportPackage`（`appId` / `schemaVersion` / `revision`）を不変の交換契約として保つことで、
  Web ↔ ネイティブ間も同じ JSON でデータを移せる。
- 「DB = 正本 / JSON = 交換形式 / 同期はアプリ外」という原則はプラットフォームを問わず維持する。
