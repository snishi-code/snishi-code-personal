# simple-ledger — UI / UX

medical 側のコードは参照せず、snishi-code 系の抽象 UI 規約に従う。実装は
`simple-ledger-src/src/ui/`、文言は `src/i18n/ja.ts`。

## 全体方針

- スプレッドシート風 UI にしない。操作入口を少なくする。
- 入力の主導線は **ホームの 収入 / 支出 / 振替 の 3 ボタンのみ**（ヘッダーに `+` は置かない）。
  補助操作はメニュー / 設定に集約。
- 白背景・控えめな線・teal を主色。角丸は控えめ（`--radius: 12px`）。
- タップ領域は **44px 以上**（`--tap: 44px`、`.icon-btn` / `.btn` / `.list__item` など）。
- **色だけに依存しない**: 状態はアイコン + 文言を併用（エラー=⚠+文、成功=✓+文、増減=記号 +/−）。
- 成功は toast。危険操作は明示確認。保存 / import / 削除は fail-closed。

## 画面構成（`src/ui/screens/`）

- **Dashboard**（初期表示・期間の財務ダッシュボード）: H1 は視覚的に期間ラベル（`periodLabel`。
  アクセシブル名は「ホーム」）。上部に **収入 / 支出 / 振替** の 3 入力ボタン（唯一の入力起点）。
  期間切替は**ヘッダー中央のコンテキスト表示＋軽量ピッカー**から（画面内にスイッチャーは置かない）。
  期間の収支（収入/支出/収支）と財政状態（資産/負債/純資産）。
  **各項目はそれぞれ専用の「内訳 + 推移」ページへ遷移する**（旧・財務諸表 PL/BS トグルを項目ごとに分解。
  同じページに集約しない）: 収入→`incomeBreakdown` / 支出→`expenseBreakdown` / 収支→`netIncome` /
  資産→`assetsBreakdown` / 負債→`liabilitiesBreakdown` / 純資産→`netAssets`。**フロー（収入/支出/収支）は
  期間合計、ストック（資産/負債/純資産）は期間末の基準日（`periodAsOf`: 月→月末 / 年→年末 /
  全体→最終データ日 or 今日）**で出す（フローとストックを混ぜない）。
  **「支出」= 通常支出 + 継続コスト**。固定資産の購入額そのもの・返済・振替は支出に含めない
  （購入は資産取得、償却分の費用＝継続コストとして計上）。**支出はホーム独立セクションにしない**。
  **収支** = 収入 − 支出（生活余剰）。
  年別 / 全体では **推移を SVG グラフで俯瞰**（`TrendChart`: 収支（=収入−支出）/ 支出=フロー
  （`variant='bar'`）/ 純資産=ストック（`variant='line'`）。年別=12ヶ月、全体=**年集約**。軸・ゼロ線・
  目盛ラベルを持つ自前 SVG（外部チャートライブラリ不使用）。全体の年ラベル（`UI.period.trendPoint`）は
  タップでその年の月別表示へドリルダウン。色だけに依存せず、目盛の数値ラベル・各点の aria-label・
  スクリーンリーダー向け要約（sr-only）を持つ）。縦長リストにはしない。投資の評価損益は財政状態に補助表示。
  下部に **期間内の仕訳**（最大 5 件・日付降順、生成仕訳も表示）と「すべて見る」（期間フィルタ付きで
  仕訳画面へ）。期間は App の共有 state（`ReportPeriod`）で、ホーム / 各内訳ページ / 仕訳に伝播する。
- **内訳ページ（収入 / 資産 / 負債 / 純資産）= `Breakdown`（汎用・`section` で切替）**: ホームの各項目の
  タップ先。「その項目の科目内訳（行タップで仕訳へドリル）+ 推移グラフ」を見せる。会計概念は内部に残し、
  画面名・見せ方は個人家計向けに柔らかくする。**フロー（収入）はドリルに期間、ストック（資産/負債/純資産）は
  基準日を引き継ぐ**。純資産は元手の科目 + **今期の損益**（= retained earnings）+ 純資産の推移を出し合計は純資産。
  負債は **資金繰り・返済計画への導線**（`liabilitiesBreakdown.cashflowLink` → `cashflow`）を持つ。
  推移は **単一正本** `breakdownData.buildSectionTrends`（ホーム・各内訳ページが同じ定義を使い数字をズラさない）。
- **ExpenseBreakdown（支出の内訳）**: ホーム「支出」のタップ先。通常支出 / 継続コスト /
  支出合計を期間に応じて表示し、支出の推移も出す。継続コストをタップすると継続コスト台帳
  （`allocations`）へ進める（CF へは飛ばさない）。
- **NetIncome（収支）**: ホーム「収支」のタップ先。収入 / 支出 / 収支の要約 + **月ごとの残り方
  （収支の推移）を主役**にする。科目別ドリルダウンはしない。
- **Journal**: 仕訳一覧。検索（摘要・メモ）・期間・**勘定科目絞り込み**（各内訳ページからの遷移）。
  行タップで編集、各行に **取消/返金**（逆仕訳）と削除。**ヘッダー期間（正本）に追従**して日付範囲が
  変わる（ユーザーがヘッダー期間を変えたとき）。ただし**初回マウントでは追従しない**＝ドリルダウンで
  渡された日付範囲（フロー=期間 / ストック=基準日）を上書きしない。日付・科目は画面内でも手動で上書きできる。
- **Entry**（シート）: 入力モード（収入/支出/振替/詳細）。日常入力は**お金の流れ `源泉 → 行き先`**で
  見せ、**借方/貸方や資産/負債/収益/費用の分類見出しは出さない**。並びは人間の入力順
  （**日付 → 項目 → 金額 → お金の流れ（A → B）→ 詳細**）。内部対応は常に source=貸方(credit) /
  destination=借方(debit): 収入 `収入元 → 入る場所`、支出 `支払い方法 → 使い道`（現金/預金/カードを
  並列に flat 表示）、振替 `移動元 → 移動先`（項目は任意・未入力なら「移動元 → 移動先」を自動付与）。
  候補は科目の **`role`** で絞る。`FlowPicker`（`AccountPicker` の flat モード + 矢印 `→`）を使う。
  メモ・タグは「詳細」折りたたみに収める。
  科目は**チップピッカー**（`AccountPicker`、単一選択 radio）。「詳細入力」で借方/貸方直接指定
  （manual）へ切替可。取消/返金は逆仕訳をプリセット。日付の既定は今日（必須）。支出では**お金の流れの
  行き先に「継続コスト化」ボタン**を置き、タップで行き先が**継続コスト対象名の自由入力欄**に変わる
  （`data-ui=journal.entry.ccToggle` / `ccName`）。入力順は**対象名 →（何か月分として見るか）月数 →
  分類先カテゴリ（費用・`ccCategory`）→ 継続・買い替え**。対象名は**勘定科目ではなく台帳項目名**
  （`MonthlyCostItem.name`・勘定科目は増えない）。**資産経由モデル**: `支払い元 → 継続コスト台帳`（資産化）/
  `継続コスト台帳 → 費用カテゴリ`（費用へ分類）を**仮想展開**する（`createContinuousCost`。実支払い仕訳は作らない）。
  支払い元は資金・カード・**ローン（other-liability）**を許可（自動車ローンで自動車を買う等）。通常支出でローンを
  支払い元にはできない（`entry.error.loanNotExpense`）。継続コスト ON ではタグ欄を隠す。支払い元が負債（カード/ローン）
  のときだけ「分割/後日引落を資金繰りに入れる」を出し、引落口座・返済回数・初回引落日を入力できる
  （返済は費用でなく CF）。固定資産科目を行き先に選ぶ旧 `fixed-asset` 経路は互換として残す。
  - **支払い元の補助導線**: 既定は現金・預金・クレジットカード。**取り置き資金は「取り置き資金を使う」トグル**
    （`reserveToggle`）を開いた時だけ支払い元候補に並ぶ（通常入力を軽く保つ）。**ローンは「ローンを組む」単一導線**
    （`liabilityToggle`＝支出のラベルは「ローンを組む」）で、開くと既存ローンが支払い元候補に並び、`liabilityCreate`
    （「新しいローンを作成」）で新規ローンも作れる。**クレジットカードは支払い方法として既定で出す**ため、ローン導線とは混ぜない
    （カード新規作成は勘定科目管理で行う）。編集時に既に取り置き/ローンが選ばれていれば、そのトグルは開いた状態で表示する。
- **継続コスト台帳**（`Allocations.tsx`、画面名は継続コスト）: 継続コスト対象を一覧。**継続購入（自動更新）と
  償却のみ（再購入なし）を小ラベルで区別**する。総額・月額・何か月分・更新周期・今月の計上額・分類先カテゴリ・
  支払い元・状態を表示。一時停止/再開・編集ができる。費用への分類は仕訳ではなく仮想展開（`derivedEntries`）で
  PL/BS/支出に反映され、集約台帳口座（『継続コスト台帳』）残高に未消化分が残る（二重計上しない）。
  - **過去から再計算の明示**: 編集シートで総額・開始月・月数・更新周期・終了月・分類先カテゴリのいずれかを
    変えると、「過去から再計算されます」の注意を出す（`data-ui=allocations.edit.impactWarning`）。資産経由
    モデルは辞書を展開し直すので過去集計が変わる＝仕様（[過去編集モデル](ledger-concept.md#過去編集モデルあとから意味を組み替える)）。
  > **旧 Statements（PL/BS トグル画面）は廃止**。個人家計簿として硬く分かりづらいため、ホームの各項目ごとの
  > 内訳ページ（上記 `Breakdown` / `ExpenseBreakdown` / `NetIncome`）に分解した。会計概念（PL/BS、当期純損益、
  > 貸借一致）は内部の導出ロジック（`accounting.ts`）に残し、ユーザー向け画面名・見せ方は柔らかい具体名に置換。
  > 用語の内部型名は `revenue`/`expense`/`netIncome`/`asset`/`liability`/`equity` のまま。**期間はヘッダーの
  > 期間メニューと共有**（`ReportPeriod`。画面内に期間スイッチャーは置かない）。フローは期間合計、ストックは
  > 期間末の基準日（`periodAsOf`）で導出し、基準日は文言で表示する。
- **Accounts**: 勘定科目を区分ごとに残高つきで管理（役割 `role` も表示）。追加 / 編集 /
  アーカイブ / 削除（参照中は不可）。シートで `type` と `role` を選ぶ（`type` 変更時は `role` を
  既定へリセット、`type` と矛盾する `role` は保存不可）。**内部集約・自動生成の役割（継続コスト台帳=
  `continuing-cost-asset`）はユーザー編集対象でないため一覧・ロール選択肢に出さない**（`isInternalRole`。
  BS / 資産内訳には残高として 1 行で現れる）。**使用中の科目は区分(type)を変更不可**
  （UI で無効化、`upsertAccount` でも fail-closed）。**`role` は使用中でも変更可**（会計残高は
  変わらず入力候補が変わるだけ）。「使用中」判定は仕訳/予定CF/取り置き資金/継続コストの参照を含む。
- **補正・勘定科目**（`Adjustments.tsx`、画面名 `manage.title`、設定の「管理」から）: 2 つの管理機能を 1 画面に。
  - **初期残高**（上部・`OpeningSection`）: 新規 BS 科目の作成 + 開始残高付与、または既存科目への付与
    （`kind='opening'`）。一覧から**編集・削除**できる。「初回設定にも使えます／あとから変更できます」。
    ホームの日常入力には初期残高導線を置かない（[初期残高](ledger-concept.md#初期残高)）。初回起動オンボーディングは別タスク。
  - **残高補正**（下部）: 新規補正フォーム（対象科目・種類・日付・実残高 → 理論残高/差額プレビュー）+
    登録済み補正の一覧。各行から**編集・削除**（[現実アンカー](ledger-concept.md#残高補正締めなし現実アンカー)）。
    編集時の理論残高は**その補正自身を除いて**計算し直す（二重掛け回避）。差額 0 は削除扱い。削除は確認必須。
    補正仕訳は Journal では読み取り専用（この画面でだけ管理）。
- **Cashflow（資金繰り）**: **確認専用の画面**（独立した入力フォームは持たない）。責務は
  **「現金が枯渇しないか」と「負債返済」を見る**こと。主役は上から ①**自由資金の推移**
  （現在/最低自由資金・枯渇警告・期間 3/6/12/24ヶ月。総資金=流動資金＝`daily/reserve-asset` のみ）→
  ②**支払用負債・返済予定**（`payment-liability` ごとに残高 + 返済予定の次回支払日/残回数/残額。**残高が
  あるのに予定が無い負債は「返済予定が未登録」と注意表示**）→ ③**未来の入出金・振替予定**（読み取り）→
  ④**分割・定期の返済予定**（読み取り）。⑤**取り置き資金・資金目標**は**下部に畳む補助情報**。
  - **入力はホームに一本化**: CF 画面に 収入/支出/振替 ボタンや予定の独立追加 UI は置かない。
    ホームの 収入/支出/振替 で**未来日付**を選んだ通常仕訳が CF に反映される（`cashDeltaOfEntry` で流動資産の
    純増減を求め、`projectCashflow` の `futureEvents` として投影に積む）。二重計上にしない: 未来仕訳は
    `today` 時点の `startTotal` に未含・予定 CF は `status==='planned'`（未実績）で互いに素。
  - **`CashflowSchedule`（予定CF）はデータとして維持**するが、**生成は継続コストの負債払い・借入（振替）の
    分割返済から自動**で行い、CF 画面では一覧 + 実績化 + 削除のみ（手動追加 UI は撤去）。`projectCashflow` は
    `transfer` では総資金・自由資金を増減させない。長期ローン専用スキーマは作らない。
- **Settings**: 先頭に **「管理」セクション**（勘定科目 / タグ / 残高補正 への遷移。
  各内訳ページはホームの各項目から辿るため含めない）。
  続いて JSON export / import、スナップショット、台帳設定（**期待年利%** を含む）、アプリ情報、全データ削除。
  期待年利は資金目標の必要積立額の参考計算にのみ使う（投資助言ではない・既定 0）。
- **Help**: 使い方の説明（モーダル）。

## ヘッダー（`src/ui/Header.tsx`）

- 左: ホームアイコン（→ Dashboard）
- 中央: **現在の期間コンテキスト表示**（操作ボタン群は常設しない）。`periodLabel` を年/月の 2 チップに分け、
  タップで対応する**軽量ピッカー**を開く。月表示=`2026年 ▾ / 6月 ▾`、年表示=`2026年 ▾ / 年全体 ▾`、
  全期間=`全期間 ▾`（月チップ無し）。data-ui は `UI.period.yearTrigger` / `UI.period.monthTrigger`。
  台帳名はヘッダーに出さない。
- 右: `≡`（メニュー）

入力は **ホームの 3 ボタンに集約**し、ヘッダーに `+` は置かない（入力の種類シートも廃止）。

### 期間の軽量ピッカー（`src/ui/PeriodPickers.tsx` / `src/ui/Popup.tsx`）

期間は「DB からの抽出条件」。ヘッダーのコンテキスト表示をタップして開く小さな選択リスト
（`Popup`: 背景タップ/Escape で閉じ、タイトル・閉じる・完了ボタンを持たない）。選んだら即 `onChange`
して閉じる。現在行は `aria-current`。

- **年ピッカー**（`PeriodYearPicker`）: 行 = `全期間`（→`all`）+ 利用可能年。年を選ぶと、現在が月表示なら
  月を保持して `month`、年/全期間なら `year`。候補は `availableYears`（データ・予定CF・資金目標・
  取り置き資金の目標日 + 現在/翌年 + 選択中。現在年±50 にクランプ）。
- **月ピッカー**（`PeriodMonthPicker`）: 行 = `年全体`（→`year`）+ `1月`〜`12月`（→`month`）。

期間は `App` の共有 state（`ReportPeriod`）でホーム / 各内訳ページ / 仕訳に伝播する（画面ごとに分岐しない）。
旧来のモーダル型期間メニュー（`PeriodMenu`/`PeriodSwitcher`・ネイティブ select）は廃止。

## メニュー（`src/ui/Menu.tsx`）

**中央モーダル / モバイルは下部シート**（共通 `Modal` の dialog variant。右ドロワー `.drawer` は廃止）。
背景タップ・Escape で閉じ、メニュー内クリックでは閉じない。**管理・補助機能に絞る**:
**継続コスト / 資金繰り / 設定 / ヘルプ**。ホーム=ヘッダーのホームアイコン、
仕訳=ホーム下部「期間内の仕訳」やドリルダウン、各内訳ページ=ホームの各項目から辿る（メニューには置かない）。
勘定科目 / タグ / 残高補正は**設定の「管理」セクション**。

## モーダル / シート規約（`src/ui/Modal.tsx`）

- 閉じ方は **`dismissMode`** で統一する（旧 `dismissable` は後方互換: `false`=`never` 相当）:
  - **`always`**: メニュー / ヘルプ / 軽い選択。背景タップ・`Escape` で閉じる（選んだら閉じる）。
  - **`if-clean`**: 入力フォーム。未編集なら閉じ、**編集済み（dirty）は破棄確認**を出す
    （`useDirtyGuard`。「入力を破棄しますか？」→ 破棄する / 編集を続ける。確認自体は `never`）。
    dirty 判定は各シートで入力フィールドのスナップショットを初期値と比較する。`×` / 背景タップ /
    `Escape` / フッターのキャンセルはすべて `requestClose` を通す。
  - **`never`**: 削除 / 全削除 / import / 復元など破壊的操作。背景タップ・`Escape` で**閉じない**。
- 背景タップ判定は **`onPointerDown` かつ `e.target === e.currentTarget`**（内部要素のタップでは閉じない）。
- 保存 / キャンセルがあるフォームは下部に明確なボタン。フォーカストラップ + 復帰、`role="dialog"` /
  `aria-modal`。

## toast（`src/ui/toast.tsx`）

- 成功 = success（teal）、失敗 = error（赤）。アイコン + 文言。`role="status"` / `aria-live="polite"`。
- **保存失敗時に成功 toast を出さない**（store の各アクションは失敗時 error toast + 例外送出）。

## 危険操作（`src/ui/ConfirmDialog.tsx`）

- 削除 / import 上書き / 復元は確認ダイアログ。確定ボタンは danger 色 + ⚠。
- 全データ削除は **キーワード入力一致**（「削除」と入力）まで確定を無効化。

## アクセシビリティ

- 主要操作は `getByRole` / `getByLabelText` で辿れるよう、ラベルと control を id で結ぶ。
- フォーム入力は 16px 以上（iOS のズーム回避）。
- コントラストは WCAG AA を満たす（teal-700 を白文字ボタン背景に使用）。
- スキップリンク、`:focus-visible` のフォーカスリング。
